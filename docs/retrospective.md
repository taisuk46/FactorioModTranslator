# Factorio Mod 自動翻訳ツール - プロジェクト振り返り

## 総評

アプリケーションは最終的にビルド・起動まで到達しましたが、**実装〜リリースの過程で多くの手戻りが発生**しました。以下に「悪かった点」を分類して洗い出します。

---

## 🔴 重大な問題

### 1. 起動しない exe を「完成」としてしまった

| 項目 | 内容 |
|---|---|
| **問題** | タスク管理上は「ビルド・動作確認 ✅」としていたが、実際にはアプリが起動しなかった |
| **根本原因** | `dotnet build` の成功＝動作確認、と誤認。**実際に exe を起動して画面表示まで確認するテストを行わなかった** |
| **影響** | ユーザーがexeを実行して初めて問題が発覚し、長時間のデバッグセッションが必要になった |

> [!CAUTION]
> ビルド成功 ≠ 動作確認。**起動→画面表示→基本操作**まで確認して初めて「動作確認完了」とすべきだった。

### 2. `PublishReadyToRun` + SingleFile の相性問題を事前に把握していなかった

| 項目 | 内容 |
|---|---|
| **問題** | `PublishReadyToRun=true` のシングルファイルビルドがサイレントクラッシュする環境依存の問題 |
| **根本原因** | WPFアプリのシングルファイル配布に関する既知の制約を事前調査しなかった |
| **影響** | v1.0 → v1.3 まで4回のリビルドが必要になった |

### 3. APIキー未設定時の起動クラッシュ（`authKey is empty`）

| 項目 | 内容 |
|---|---|
| **問題** | DeepL SDK は空文字列の API キーを渡すとコンストラクタで即座に例外をスローする |
| **根本原因** | 初回起動時（APIキー未設定）のシナリオをテストしていなかった |
| **影響** | ユーザーの初回起動が必ず失敗する致命的バグ |

> [!WARNING]
> 外部SDKのコンストラクタが例外を投げるケースは、**遅延初期化パターン**で防御すべきだった。

## 🟠 設計・アーキテクチャの問題

### 4. DI（依存関係注入）の不安定な実装

| 項目 | 内容 |
|---|---|
| **問題** | DIの実装が `App.xaml.cs` の `OnStartup` → `MainWindow.xaml.cs` のコンストラクタへ何度も変更された |
| **根本原因** | WPFの `StartupUri` と手動起動（`OnStartup` でのインスタンス管理）の使い分けを最初に決定しなかった |
| **最終状態** | `MainWindow.xaml.cs` のコンストラクタ内で全サービスを直接 `new` している（DIコンテナ未使用） |
| **改善案** | `Microsoft.Extensions.DependencyInjection` を使い、`App.xaml.cs` で `IServiceProvider` を構築すべき |

### 5. テストが実質的に機能していない

| 項目 | 内容 |
|---|---|
| **問題** | タスク管理上は「ユニットテスト ✅」だが、テストが実際に実行・パスしたかの確認記録がない |
| **根本原因** | テストファイルは作成したが、`dotnet test` を実行して結果を確認するフローが省略された |
| **現状** | `CfgParserTests.cs`, `ModLoaderTests.cs` は存在するが、テストの実行結果が不明 |

### 6. 翻訳エンジンの切り替えが未実装

| 項目 | 内容 |
|---|---|
| **問題** | 設定画面でエンジン（DeepL/Google）を選択できるUIはあるが、実際の切り替えロジックが未実装 |
| **根本原因** | `MainWindow.xaml.cs` で `DeepLTranslationEngine` をハードコードしている |
| **改善案** | 設定変更時にエンジンインスタンスを再生成する仕組み、またはファクトリーパターンが必要 |

---

## 🟡 プロセス・品質面の問題

### 7. ビルドエラーの連鎖的な修正

| 項目 | 内容 |
|---|---|
| **問題** | XAMLの `&` エスケープ漏れ、`GlossaryInfo` の型名不一致、`using` の不足など、ビルドエラーが連鎖的に発生 |
| **根本原因** | 大量のファイルを一気に作成した後にまとめてビルドしたため、エラーが蓄積された |
| **改善案** | **1ファイル作成ごとにビルド確認する**インクリメンタルな開発フローを採用すべき |

### 8. デバッグ用ファイルの残存

| 項目 | 内容 |
|---|---|
| **問題** | `startup_debug.log`, `error.txt`, 4つの `publish_*` フォルダ、複数の旧バージョンZIPが残っている |
| **根本原因** | デバッグ完了後のクリーンアップ手順が定義されていなかった |
| **現在残っているゴミ** | `startup_debug.log`, `error.txt`, `publish_normal/`, `publish_test/`, `publish_final/`, `v1.0〜v1.2.zip`, `UnitTest1.cs`（テンプレファイル） |

### 9. `.gitignore` が未作成

| 項目 | 内容 |
|---|---|
| **問題** | 実装計画には `.gitignore` の作成が含まれていたが、実際には作成されていない |
| **影響** | GitHub公開時に `bin/`, `obj/`, `publish/` 等の不要ファイルがコミットされる |

## ✅ 次回への改善アクション

1. **ビルド→起動→基本操作の動作確認を毎フェーズ末に実施する**
2. **外部SDK初期化時のエラーケース（空キー、ネットワーク不通等）を最初にテストする**
3. **インクリメンタルビルド**：ファイル1〜2個作成ごとにビルド確認
4. **DIコンテナの導入**：`Microsoft.Extensions.DependencyInjection` を最初から採用
5. **`dotnet test` の実行結果をログに残す**
6. **リリース前チェックリスト**：`.gitignore`、不要ファイル削除、全テスト実行
7. **WPF SingleFile 配布の既知の制約を事前調査する**
